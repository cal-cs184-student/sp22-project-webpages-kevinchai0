Task 1: Drawing Single-Color Triangles

In this first task, we were asked to rasterize single-colored triangles.

In order to rasterize the triangles, we first needed to determine where our starting and ending points were. We decided to start at the top left and go end up at the bottom right. We basically used the Point-in-Triangle test to determine if we should rasterize a certain point or not and did this until we reached our ending point.

Our algorithm is no worse than one that checks each sample within the bounding box of the triangle because that’s exactly what our code does. Our code determines whether or not each simple is within our bounding box given by the 3 lines created from the 3 pairs of points. We used the Point-In-Triangle test to determine if the sample was inside our bounding box.

Data Structures

We also created the bounds for the outer for loops in rasterize_triangle using two float arrays. The first one represents the top left corner of the triangle and the bottom represents the bottom right corner.

float start[] = { max(float(0.0), min(x0, min(x1, x2))), max(float(0.0), min(y0, min(y1, y2)))};
float end[] = {min((float)width, max(x0, max(x1, x2))), min((float)height, max(y0, max(y1, y2)))};

The last data structure that we used was a float array to hold the point that we are currently sampling at.

float pt[] = {(float)x + float(i * 1.0/sqrt(sample_rate)) + float(1.0/sqrt(sample_rate * 2)) , (float)y + float(j * 1.0/sqrt(sample_rate)) + float(1.0/sqrt(sample_rate * 2))};

Rasterizing Algorithm

In order to determine a triangle’s winding direction (clockwise or counterclockwise), we summed up the three sides (used Shoelace formula) to re-order the coordinates to a clockwise direction if necessary.

    float sum = (x1 - x0) * (y0 + y1) + (x2 - x1) * (y1 + y2) + (x0 - x2) * (y0 + y2);
    if (sum < 0) {
        swap(x1, x2); swap(y1, y2);
    }

The rest of our algorithm relies on the point-in-triangle test as described above, checking the side in clockwise order. Then, we add to our sample_buffer at the end – math in sample_buffer is described in task 2 portion of write-up.

    for (int x = floor(start[0]); floor(x) <= floor(end[0]); x++) {
        for (int y = floor(start[1]); floor(y) <= floor(end[1]); y++) {
            for (int i = 0; i < sqrt(sample_rate); i++) {
                for (int j = 0; j < sqrt(sample_rate); j++) {
                    float pt[] = {(float)x + float(i * 1.0/sqrt(sample_rate)) + float(1.0/sqrt(sample_rate * 2)) , (float)y + float(j * 1.0/sqrt(sample_rate)) + float(1.0/sqrt(sample_rate * 2))};

Task 2: Antialiasing by Supersampling

Supersampling Algorithm

Supersampling is useful because it alleviates some of the sampling artifacts - aka “aliasing” - in graphics. This includes jaggies, wagon wheel, etc. The supersampling allows us to have intermediate valued colors that create a gradient to allow for better transitions between the object and the background.

We used supersampling to antialias our triangles by increasing the amount of samples per (x,y) value. Then we took these values and averaged them to get a intermediate color value for the point when we downsampled. The supersampling is handled in rasterize_triangle and the downsampling is handled in resolve_to_framebuffer.

Data Structures

We resized the sample buffer to hold the supersampled colors so that we could later add to rgb_framebuffer_target

sample_buffer -> size = width * height * sample_rate

Algorithm

The modifications that we made to the rasterization pipeline include adding a nested for loop on the inside of our current loop.

for (int i = 0; i < sqrt(sample_rate); i++) {
    for (int j = 0; j < sqrt(sample_rate); j++) {
        float pt[] = {(float)x + float(i * 1.0/sqrt(sample_rate)) + float(1.0/sqrt(sample_rate * 2)) , (float)y + float(j * 1.0/sqrt(sample_rate)) + float(1.0/sqrt(sample_rate * 2))};
        ...
        sample_buffer[(y * (int)sqrt(sample_rate) + j) * width * (int)sqrt(sample_rate) + x * (int)sqrt(sample_rate) + i] = color;
        }
    }

This nested loop contains the same point-in-triangle check that we used before, the only difference being that now we need to do some extra math in order to account for the different sampling spots in each pixel.

The offset that we used for both the x and y is float([i/j] * 1.0/sqrt(sample_rate)) + float(1.0/sqrt(sample_rate * 2). This offset accounts for which sub-pixel we are located in (using i and j), and then offsets to the center of that pixel with the second square root portion.

One of the challenges that we came across was figuring out how to store this information inside of sample_buffer and retrieve it later to downsample. What helped our group out was drawing a sample diagram of the pixels and how we were going to store them.

This allowed us to come up with the new formula sample_buffer[(y * (int)sqrt(sample_rate) + j) * width * (int)sqrt(sample_rate) + x * (int)sqrt(sample_rate) + i] = color; for storing the colors.

Task 4: Barycentric coordinates!

This task involved implementing baycentric interpolation.

We first had to find the baycentric coordinates of our triangle. Baycentric coordinates are coordinates that are determined by the weights of the vertices of a triangle. Normally the baycentric coordinates are inside the triangle if the vertices are all non-negative. The baycentric coordinate system is represented in alpha, beta, and gamma values which sum to 1. The below image shows how we can convert from a 2D-coordinate system to baycentric one.

The following example explains baycentric coordinates more visually. We give each vertex a color (RED, GREEN, BLUE). Then the rest of the triangle can be colored by calculating the baycentric coordinates and interpolating.

The reason why we use Barycentric coordinates as opposed to Cartesian is that we can define a point with respect to n+1 other points instead of just with respect to the origin. This allows us to more easily apply gradients of colors and textures, as we can move within respect to vertices of our triangle in this case.

Task 5: “Pixel sampling” for texture mapping

In this task, we implemented texture sampling using nearest neighbor and bilinear interpolation. We are applying a 2D texture to a surface which is 2D or 3D. This allows us to create depth perception, shadows, reflections, projections, and a lot of other visual effects for our texture.

Pixel sampling allows us to apply textures to images. We implemented it by calculating the (u,v) coordinates using our baycentric coordinates. After getting the (u,v) coordinates, we used either the nearest neighbor or bilinear pixel sampling methods to determine the texture color.

The nearest neighbor sampling method takes in the (u,v) coordinates and then returns the texel closest to those coordinates.

Bilinear sampling is a little more complicated. Bilinear sampling takes in the (u,v) coordinates and then takes the 4 nearest texels and performs interpolation to return the average texel color. We followed the slide below to perform the lerps. We first did two linear interpolations horizontally and then one vertically to get our weighted average texel color of the 4 texels.

We can notice in the examples below that bilinear sampling defeats nearest sampling. The edges are more smooth when lookng at the top of the Campanile using bilinear sampling. This makes sense because bilinear sampling takes more samples (4) than nearest sampling (1) meaning that the image created used bilinear sampling will be more accurate.

Since nearest sampling only takes one sample to determine the texture color, bilinear sampling will generally perform better on images with higher frequency. This is because nearest sampling will tend to miss out on some data of the image whereas bilinear sampling does a better job at maintaing the data.
