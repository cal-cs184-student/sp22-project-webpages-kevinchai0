Section I: Bezier Curves and Surfaces

Part 1: Bezier curves with 1D de Casteljau subdivision

De Casteljau’s algorithm is a way to evaluate Bezier Curves. The algorithm takes in n points and recursively interpolating points and creating new line segments. The algorithm stops when we cannot create any line segments as we have no more control points left. Then, we are left with a point that lies on the Bezier Curve.

This algorithm relies on a ratio t (ranging from 0 to 1) that determines how far we evaluate the point on each line segment.

Our implementation evaluated one step of the de Casteljau’s algorithm using given points and the value of t. We followed the following formula to calculate either the next step (vector of intermediate points) or the final interpolated vector.

Part 2: Bezier surfaces with separable 1D de Casteljau subdivision

This part involved extending de Casteljau’s algorithm to Bezier surfaces. Bezier surfaces are parametric surfaces defined by a 2D grid of control points.

In order to accomplish this, we first had to evaluate point u for the 4 Bezier curves in the u dimension. We then used de Casteljau’s to evaluate point v and return a point that lies on the Bezier surface.

To be more specific: we are given an n x n grid of control points and the parameters u, v. Given these control points, we can define a Bezier curve using u and each row of control points. Using 1D de Casteljau’s we can evaluate a single point on this Bezier curve, P1. Then using P1, we can use 1D de Casteljau find a single point P that lies on the Bezier Surface.

To implement this, we first had to implement the following functions:

evaluateStep (3D): evaluates a single step of de Casteljau's for points in 3D
evaluate1D: fully evaluates de Casteljau's for points in 3D
evaluate: evaluates Bezier patch given parameters u and v

Using these functions, we called evaluate1D on each row of control points using u (the size of our control points). Then we called evaluate1D again on the result of the previous calls using v.

Part 3: Average normals for half-edge meshes

In this part, we implemented Phong shading using vector normals which provides better shading for smooth surfaces. In order to implement area-weighted normal vectors at vertices we first looked at printNeighborPositions(...) to understand how to iterate through faces incident to a vertex. We travesed through all of the neighboring triangles and used face->normal() to calculate the normal for each face. We added all of the normals to a Vector3D that we intialized with all zeroes. At the end, we returned the unit vector of our Vector3D. Since we used face->normal() we had to ensure that we handled boundaries correctly and skipped over them.

    Vector3D N( 0., 0., 0. );

    HalfedgeCIter h = halfedge();      // get the outgoing half-edge of the vertex
    do {
      HalfedgeCIter h_twin = h->twin(); // get the opposite half-edge
      VertexCIter v = h_twin->vertex(); // vertex is the 'source' of the half-edge, so
      if (!h->isBoundary()) {
        VertexCIter v = h_twin->vertex();
        N += h->face()->normal();
      }
      h = h_twin->next();               // move to the next outgoing half-edge of the vertex
    } while(h != halfedge());          // keep going until we are back where we were
    return N.unit();

If we didn’t use face->normal(), we would compute the normal of each face by getting the cross product of two vectors along the face. We would able to get all the vertices of the triangle to get these two vectors by using ->next(). We would then add the results of the cross product to our Vector3D and return the unit vector of Vector3D. We don’t have to check for boundaries because the other method is meant to deal with faces that aren’t planar. We included the code for both solutions in the repository.

    Vector3D N( 0., 0., 0. );
    HalfedgeCIter h = halfedge();      // get the outgoing half-edge of the vertex
    Vector3D center = h->vertex()->position;
    do {
      h = h->twin(); // get the opposite half-edge
      Vector3D p1 = h->vertex()->position;
      Vector3D v1 = p1 - center;
      h = h->next()->twin();
      Vector3D p2 = h->vertex()->position;
      Vector3D v2 = p2 - center;
      N += cross(v2, v1);
      h = h->twin();               // move to the next outgoing half-edge of the vertex
    } while(h != halfedge());          // keep going until we are back where we were
    cout << N.unit() << endl;
    return N.unit();

Sharp corners and edges

With sharp corners, the degree of the corner drastically affects how it will be upsampled. If the degree is higher, it will become a sharper corner. If the degree is lesser, it will get smoothened out to a curve. We can reduce the effect of smoothening everything out by splitting the corners before upsampling. This can be seen in the images below.

We can reduce the effects of asymmetry by splitting the edges evenly. Otherwise, we can end up with images like the last two, where the original mesh looked fine but after upsampling the vertex with the higher degree sharpened compared to the rest of the image.

Noe, we can pre-process cube.dae by splitting each of the edges on the faces to increase its symmetry. This way, when we split and flip the edges later they won’t be skewed toward one direction.

As you can see, adding 6 split edges allowed us to change the upsampling of the mesh drastically. This occurs because now each corner of the cube is splitting at the same rate, which will allow us to have the same upsampling effects on each corner. We made all of the corners of the cube have the same degree, which is critical as to how they’re going to split in the upsampling.

Pre-processing alleviates these skewing effects by making sure that the model is more symmetric before splitting, as any asymmetries will be emphasized when upsampling, resulting in some corners of the image being more “smoothened out” by the upsampling and others being sharper.
